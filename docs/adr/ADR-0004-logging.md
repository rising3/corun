## 概要

運用・デバッグを容易にするため、ログの形式と必須フィールドを標準化する。

## コンテキスト

解析と集計を自動化できる構造化ログが望まれる。標準がないと、分析やモニタリングの導入コストが増加する。

## 決定

ログ出力は機械で解析しやすい JSON 形式を推奨し、各実行ログに以下のフィールドを含める:

- `timestamp` (ISO8601)
- `spec_id` (該当する仕様/コマンドの識別子)
- `version` (仕様/コマンドのバージョン)
- `command` (実行されたコマンドと引数)
- `exit_code` (プロセス終了コード)

## 決定の理由（Rationale）

- 構造化ログは検索・集計・アラート設定が容易で、自動化ツールと相性が良い。

## 影響とフォローアップ

- `stdout` は主要な結果を JSON で出力し、`stderr` は診断情報に利用する方針とする。
- ログスキーマのバージョン管理とサンプルログを `docs/` に追加すること。

## 代替案

- テキストログの維持は可能だが、解析効率の観点で推奨しない。
